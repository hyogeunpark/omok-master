# 오목 (Omok) — 프로젝트 인덱스 / 마스터 스펙

> Spec-Driven Development의 진입점(entry point). 프로젝트 전반과 **모든 Phase에 공통으로 적용되는 규칙**을 정의한다.
> 각 단계의 구체 요구사항은 `spec/phase-*.md`, 기술 구조는 `architecture.md` 참조.

---

## 1. 개요

- **제품**: CPU와 1:1로 대결하는 오목(Gomoku) 게임.
- **목표 규칙**: 렌주 룰(흑 금수) + **타라구치-10(Taraguchi-10)** 오프닝(스왑) 규칙.
- **플랫폼**: 웹 우선(React) → PWA 설치 → 추후 Capacitor 네이티브화.
- **백엔드**: 없음. 모든 로직은 클라이언트에서 실행. 서버/계정/온라인 대전 없음.
- **방식**: 단계적 개발(Phase 1 → 2 → 3). 각 단계는 이전 단계 코드를 재사용하고 그 위에 규칙을 얹는다.

## 4. 공통 규칙 (모든 Phase 공통 기반)

> Phase별 문서는 이 절을 전제로 한다. 각 Phase는 여기에 규칙을 **추가/제약**할 뿐 이 기반을 바꾸지 않는다.

- 보드: 15 × 15 격자. 돌은 **선의 교점**에 놓는다(총 225 교점).
- 흑 선공, 이후 흑·백 번갈아 1수씩 착수.
- 가로·세로·대각선 중 한 방향으로 **5목 연속**이면 승리. (Phase 2에서 흑 장목 예외 추가)
- 빈 교점에만 착수 가능. 점유/비차례/게임종료 시 입력 무시.
- 보드가 모두 채워지고 승자가 없으면 무승부.

## 5. 공통 비기능 요구사항 (Non-Functional)

- **반응형**: 모바일 세로 화면 우선, 보드는 화면 폭에 맞춰 정사각 비율 유지.
- **오프라인**: PWA 설치 가능, 네트워크 없이 동작.
- **무료 호스팅**: 정적 배포(GitHub Pages 등), 서버 비용 0.
- **성능**: CPU 1수 응답이 보통 난이도에서 체감 지연 없이(수백 ms 이내) 완료.
- **이식성**: 게임 로직(ENGINE/AI)은 UI/프레임워크 비의존 순수 JS로 유지.
- **접근성**: 색 외 마커로도 마지막 수/금수를 구분 가능하게.

## 7. 용어 (Glossary)

- **금수(禁手)**: 흑만 둘 수 없는 반칙 수(3-3, 4-4, 장목).
- **열린 3 / 열린 4**: 양끝이 막히지 않아 다음 수로 더 강한 형으로 발전 가능한 형태.
- **장목(長目)**: 6목 이상 연속. 흑에겐 금수, 백에겐 승리.
- **스왑(swap)**: 오프닝 절차 중 색(흑/백)을 교환하는 권리/행위.
- **잠정 색**: 오프닝 종료 전 swap으로 바뀔 수 있는 임시 흑/백 배정.

## 8. SDD 사용 규칙

- 이 문서들이 **진실의 원천(source of truth)**. 규칙/요구사항 변경은 **코드보다 문서를 먼저** 수정한다.
- 구현 지시 예: *"index.md와 architecture.md를 전제로, spec/phase-2.md의 금수 판정을 `src/engine/forbidden.js`에 구현해줘."*
- 각 Phase는 자신의 **완료 기준(acceptance)**을 모두 만족해야 다음 Phase로 넘어간다.
